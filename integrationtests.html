Integration Tests

Build Phases

This system utilizes a three layered version of the N-Tier architecture. Here, lower layers are providing the functionality to support layers above them. This architecture creates a dependency amongst the components where the lower layered components need to be built before the ones in higher layers. Given these conditions, we will divide the system build into three distinct phases, each corresponding to a layer according to the system architecture.

Dividing the testing process into different phases will help with the overall process and allow it to build and mature along with the system. Additionally, it will allow for automation opportunities which can lead to continuous integration, where all changes are integrated immediately using automated pipeline and is built, packaged and tested. This ensures integration issues are caught and resolved early in the development process.

Following this approach, we will start our test process from the lowest most layer which is Database or data access layer, moving to application programming interface layer and finally proceeding to presentation or graphical user interface layer. Since we are following the path of dependency between these layers, we will be able to solidify the foundations for higher layers by testing the lower layers first. These will be the three build phases we will test:

  • Data Access Layer: Within Data access layer, we will focus on testing data access objects, ensuring they are functioning as intended. We would also focus on testing security & performance aspects of the database.
  • API Layer: Testing efforts will be focussed on security testing as well as functional testing to ensure APIs are functioning per specification and are secure. Since APIs will be exposed to external world via a programming interface, more focus on security is needed.
  • Presentation Layer: Within Presentation layer, we will focus on functional and accessibility aspects of the GUI. We would also be testing the performance of the system to ensure it complies with the non-functional requirements.
