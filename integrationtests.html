Integration Tests

Build Phases

This system utilizes a three layered version of the N-Tier architecture. Here, lower layers are providing the functionality to support layers above them. This architecture creates a dependency amongst the components where the lower layered components need to be built before the ones in higher layers. Given these conditions, we will divide the system build into three distinct phases, each corresponding to a layer according to the system architecture.

Dividing the testing process into different phases will help with the overall process and allow it to build and mature along with the system. Additionally, it will allow for automation opportunities which can lead to continuous integration, where all changes are integrated immediately using automated pipeline and is built, packaged and tested. This ensures integration issues are caught and resolved early in the development process.

Following this approach, we will start our test process from the lowest most layer which is Database or data access layer, moving to application programming interface layer and finally proceeding to presentation or graphical user interface layer. Since we are following the path of dependency between these layers, we will be able to solidify the foundations for higher layers by testing the lower layers first. These will be the three build phases we will test:

  • Data Access Layer: Within Data access layer, we will focus on testing data access objects, ensuring they are functioning as intended. We would also focus on testing security & performance aspects of the database.
  • API Layer: Testing efforts will be focussed on security testing as well as functional testing to ensure APIs are functioning per specification and are secure. Since APIs will be exposed to external world via a programming interface, more focus on security is needed.
  • Presentation Layer: Within Presentation layer, we will focus on functional and accessibility aspects of the GUI. We would also be testing the performance of the system to ensure it complies with the non-functional requirements.

Data Access Layer


Test Procedure

Integration testing data access layer is very important, partly due to the fact that most of the unit tests do not cover database as it is considered an external system and with unit tests we want them to be as much independent as possible so they can be executed while external components may or may not have been built yet. This is an important consideration since it creates the possibility of discovering significant defects which may not have been uncovered yet due to lack of unit testing. Additionally, since data access layer is the bottom most layer, other layers are inherently dependent on proper functioning of data access layer. Any lingering defects could become very hard to investigate post integration with high layers.

We have two different components here that need to be tested as part of the integration testing, with database being the more complex component and hence most of our effort will be spend testing database during this phase. The core theme of all our test cases will be around these four facts:

  • Data Mapping: We need to ensure the fields within the data access layer are mapped consistently with the corresponding fields in the database.
  • Data Integrity: Data should be consistent with every database transaction.
  • ACID Properties: It stands for Atomicity, Consistency, Isolation, and Durability. Every database transaction should confirm to these four properties.

Schema Testing

The goal of schema testing is to ensure that the mapping between the data access layer and database are same. Thus, we may also refer to schema testing as mapping testing.

Queries can be written to extract the schema and ensure it has been defined correctly and according to specification. Additionally, tools such as Ant may be used for automated testingand validation.

FTC.2

Stored Procedure Testing

Stored procedures are the user defined functions inside the database which usually execute a series of DML commands within the context of a transaction.

Stored procedures can be tested using white box testing where every single stored procedures is executed using a set of input values and results are validated against the expected values. Edge case testing is crucial.

Output values should match the expected values for each stored procedure.
FTC.3

Database Definition Testing

Database Definition testing is to ensure that all tables and fields inside the database are defined using appropriate data types with correct lengths, value validations are properly defined, foreign key constraints are correctly defined, keys and indexes are defined correctly as well.

Queries can be written against the database to extract the schema objects which can be compared to the specification against the Entity Relationship Diagram.

Output values should match the definitions in Entity Relationship Diagram.

Security Testing

Security Testing ensures that database is only accessible by authorized personnel's and level of access is appropriate/minimum required. We also ensure the database roles are correctly defined and users are allocated authorized roles only.

Queries can be written against the database to extract the roles and users which can be compared another source for validation.

Roles, users and accesses defined in the database should match specification.

API Layer


Test Procedure

Our next build focusses on API layer. This layer houses all the business logic and thus is critical in validation of the functional requirements. In fact, most of the functional requirements are also partially testable programmatically via the APIs. Every test case executed against the api should be verified at minimum against the specification for following:

  • Correct HTTP Status Code
  • Correct HTTP Response Headers
  • Correct HTTP Response Payload


During this build, we will test focus on following testing methodologies as part of our API testing strategy:

  • Functional Testing: To ensure all functionality has been implemented as intended
  • Security Testing: To ensure authentication and authorization has been properly implemented
  • Performance Testing: To ensure we meet the performance SLAs as specified
  • Load Testing: To ensure we continue to function within SLAs when under load


Test Cases

We have split the test cases for API testing according to the goal of a particular strategy. This helps us to organize our test cases and execute them together.

Security Testing

With security testing, we will try to ensure that the api's are secure and role based with authentication and authorization implemented as expected. No user should be able to consume apis without access or proper level of authorization.

Test Case # Test Case Description Expected Results

Security Principles

Ensure that the API is designed according to correct security principles. We perform both positive and negative testing to ensure persons with margin-right credentials and authentication mechanism are able to consue APIs while others are not:

Positive - Ensure API responds to correct authorization according to specification. Negative - Ensure API refuses all unauthorized calls.

Presentation Layer


Test Procedure

Presentation layer enables users to interact with the system and for this reason we will focus our testing efforts to ensure that the accessibility, usability aspects of the system are build to the specification and requirements. On a high level, we will focus on following aspects:

  • Visual Design
  • Functionality
  • Performance
  • Usability
  • Security
