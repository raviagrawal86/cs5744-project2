Test Scope

Test Items


Following system components and their related features are considered to be in scope for testing.

  • Web Application is the user facing component and will need to be test for GUI testing and will include various aspects of GUI such as usability, functionality, accessibility etc.

  • Rest API allows customers and 3rd party applications to interact programatically with the system. API's should be tested functionally as well for security aspects.

  • Database is an underlying component not exposed to user. Testing focus for database should be ensure it performs well under load testing. Additionally, security testing should be performed to ensure data cannot leak and sql injection are not possible.

  • File Storage should use testing strategy similar to database.


Following system components are considered out of scope for testing.

  • Desktop Application is out of scope being a future enhancement.

  • Mobile API is out of scope being a future enhancement.

Functional Requirements

Req# Requirement Description In Scope? Testing Goal/Description
FR-1 Pet owners (Personal Users) may self-register for an account. Yes User Accounts should be successfully created from the web application
FR-2 Veterinarian office and adoption clinic owners/administrators (Business Users) may request business accounts. Yes A business account can be created using Super Admin credentials from the web application.
FR-3 Business users may create additional accounts for employees. Yes From web application using business user credentials, an employee account can be created.
FR-4 Business users may manage roles/permissions for employee accounts. Yes From web application using business user credentials, access for an employee account can be modified.
FR-5 All users may login to the system using a unique username and password. Yes Access to web application may only be granted via proper authentication.
FR-6 Business users and employees may create or modify medical records for pets in the system. Yes User with Business and user credetials are able to define new pet records.
FR-7 Personal users may upload supporting documentation for their pets (e.g. scanned or electronic copies of medical services). Yes System must be able to store electronic documentation and tie it to a pet record. These documents must also be retrievable.
FR-8 Personal users may update/modify non-medical information of pets. Yes Using web application and valid user credentials, pet record(s) can be modified.
FR-9 Any user may pass ownership of an owned pet to any other user. Yes Testing to ensure owner of a pet can be changed from one user account to another.
FR-10 Personal users may opt in/out sharing contact info to other owners if pet is passed. No Privacy settings are not in scope for testing.
FR-11 System should notify pet owners of missing or incomplete medical or personal (e.g. vaccines not up-to-date). No External notifications will not be in scope for testing.

Non-Functional Requirements

Req# Requirement Description In Scope? Testing Goal
NFR-1 The application interface should meet or exceed level AA conformance of the Web Content Accessibility Guidelines (WCAG) 2.1. Yes Testing for Accessibility features should be done using an agreed upon 3rd party testing tool such as DynoMapper or A11Y etc.
NFR-2 The full application stack should be cloud compatible. No This non-functional requirement is a design guideline and is not in scope for testing.
NFR-3 The full application stack should autoscale to meet user demand. Yes Load Testing should be performed on the application to ensure no failure/unexpected outcomes are observed.
NFR-4 The application should be available 99.99% of the time (“four nines”). Yes Load Testing should be performed on the application to ensure no failure/unexpected outcomes are observed. An agreed upon 3rd party application performance monitoring tool such as Dynatrace, AppDynamics should be used to gather application availability metrics.
NFR-5 All web pages should load in under two seconds. Yes Load Testing should be performed on the application to ensure no failure/unexpected outcomes are observed. An agreed upon 3rd party application performance monitoring tool such as Dynatrace, AppDynamics should be used to monitor page load times.
NFR-6 All web traffic should be encrypted using Transport Layer Security (TLS). Yes Vulnerability Scanning & Penetration testing should be performed to ensure all inbound and outbound communications are secured.
NFR-7 Appropriate measures should be taken to ensure data servers are secured. Yes Penetration testing should be performed to ensure no unauthorized connection to database can be made.
NFR-8 Data backups should be created daily (minimum). No This is an operational aspect and will be out of scope for testing.
NFR-9/td> Data backups should be securely stored separately NRom data servers. No This is an operational aspect and will be out of scope for testing.
NFR-10 A private REST API should be available for third-party, desktop, or mobile application integration. Yes Rest API's should be tested functionally to ensure all operations are performing as expected. Additionally, Vulnerability testing and Penetration testing should also be performed to ensure API connections are secured.
NFR-11 Access to all data should take a least privilege approach. Yes All roles should be tested against activities requiring elevated roles to ensure role authorization has been implemented correctly.
