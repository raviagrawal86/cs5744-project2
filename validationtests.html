Non-Functional Requirements

These Non-Functional requirements impose constraints under which the system we are developing has to function. Following are the requirements that we are explicitly defining and hold true for each customer instance.

Non-Functional Requirement Description
NFR.1

Performance

We will measure the performance of our system at component level and measure of performance would be 95 Percentile Response Time. They apply to each individual user action, i.e. clicking on a button, page load etc.

  • Feedback Widget: 200ms
  • Admin Portal: 200ms
  • API’s: 500ms
NFR.2

Scalability

Our architecture and design of the system should allow to support horizontal scaling. Horizontal scaling will allow the use modern conterization technologies effectively. Each individual component should be independently scalable, on demand. One important constraint we will define is: Performance should increase as linear function or near linear function of Scale of deployment.

Following are additional scalability requirements defining the minimum criteria:

  • System must be able to handle 10% transactional growth annually
  • Data stores must be able to handle 30% of data volume/transactional growth annually
  • API’s: 500ms
NFR.3

Availability

We will measure availability as Average Uptime. Average Uptime can be measured as following:

Average Availability = Total Uptime in Minutes / Total Uptime in Minutes + Total Downtime in Minutes

We will commit to a 99.99% availability every month i.e. we cannot be down for more than 432 minutes every month including planned downtime for upgrades and unplanned system outages

NFR.4

Reliability

Reliability is the ability of a system to carry out its functions as intended and without errors.

We will measure reliability as the percentage of successful attempts (operations) compared to all attempts.

Reliability = 100% * (All Attempts – All failures) / All Attempts

We will commit to 99% reliability i.e. Only one transaction in a hundred can fail

NFR.5

Security

These are some basic guidelines that should be serve as umbrella requirements for ensuring security. However, by no means these are a comprehensive set of requirements and should be augmented by additional requirements as needed

  • All external touchpoints should have robust industry grade authentication built into them
  • All communication should use HTTPS and TLS 1.2
  • All Sensitive datastores should be encrypted
  • Incoming feedback should be scanned to ensure PHI/PII is not included
  • All web applications will use session management to ensure the integrity of a session and explicitly set expiry
NFR.6

Maintainability

  • Data stores should have archiving/purging process
  • All components should log details of failures to aid troubleshooting
  • Log files should be rotated. 15 days worth of logs should be maintained for critical components and 7 days for non-critical components
  • All changes performed to Widgets/Portal settings should be auditable and traceable
  • Production components should have limited access to individuals (minimum necessary) and all changes should be done via System
NFR.7

Compatibility

  • All user facing components should be compatible with all modern browsers including but not limited to Chrome, Firefox, Edge with version support to ensure 90% of users are supported
  • All user facing components should have response design to ensure visible aspects are not compromised when viewed from a mobile browser
  • API should use rest architecture to ensure easier utilization and greater compatibility

NFR.8

Resilience

To ensure resilience, we will ensure all components have redundancy and failover mechanism built into them. This will ensure no component acts as a single point of failure. Following points should serve as minimum guidelines:

  • All datastores should use active replication with two or more nodes in the cluster
  • All web based components should be made available via active load balancing and should have at least three or more nodes serving the requests
  • Content Delivery Network (CDN) should also be behind load balancers with at least two active nodes
  • Actives nodes for all components should be split into different geographical locations/data centers
  • All middleware components including caches and message queue should have failover nodes as well
